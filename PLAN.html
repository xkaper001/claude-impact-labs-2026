01

The failure we are actually fixing

Plenty of teams will build "a missing-person app." That misreads who the at-risk person is. We anchor on three uncomfortable truths about Kumbh and rural India — they decide the whole design.

The missing person usually cannot help themselves. The largest group is the 61–70 age band (and many older). Records say it plainly: "cannot remember name," "hard of hearing," "cannot speak properly," "keeps asking for Ramkund." They will not pull out a phone. Many never owned one. A solution that starts with the lost person's smartphone helps almost no one who is actually lost.
The family searching is panicked, often non-literate, and speaks another language. Hindi, Kannada, Bengali, Maithili, Telugu, Tamil, Marathi… spread almost evenly. They go counter-by-counter, on foot, describing a person in their own tongue to a volunteer who may not share it.
The infrastructure that exists is human, not digital. The data hands us the real assets: kendras (lost-found centers), 14 police stations, 1,280 CCTV cameras across 32 zones, and 85 chokepoints / transfer nodes / parking belts — exactly where crowds crush and families split. Networks collapse near the ghats on snan days. So the system must ride on the volunteer at the desk and survive with no signal.

One-line thesis

Build the shared spine the kendras lack: an offline-first registry where any "lost" report and any "found" person are matched across all centers by a tolerant matcher that handles missing names, wrong spellings, and 11 languages — so a reunion that takes days of walking happens in minutes.

02

What we build: Setu

An operator-mediated kiosk + a shared matching brain. Three surfaces, one core.

① Volunteer Console

A tablet/laptop PWA at every kendra. Two big buttons: I lost someone / I found someone. Voice-first intake in the family's language, auto-transcribed & translated. Photo capture if a found person is present.

offline PWAvoice + photomultilingual

② Match Engine

Every new record is scored against the opposite pool (lost ↔ found) across all centers. Tolerant of blanks, phonetic name variants, age-band, gender, language, last-seen zone & time window. Surfaces a ranked shortlist with reasons.

fuzzy / phoneticcross-centerhuman confirms

③ Ops Map

Live map of cases over the 32 zones, CCTV coverage, chokepoints & nearest police station. Shows where separations cluster so help-points and announcements can be pushed before the surge.

What the score is actually made of

  • Phonetic / fuzzy name (Soundex-style + edit distance) so Ram, Raam, Rama collapse together; skipped gracefully when blank.
  • Hard filters gender + age-band overlap — cheap, cuts the pool fast.
  • Spatio-temporal plausibility last-seen zone vs found zone, walkable in the time gap (we have all 32 zone centroids + chokepoints).
  • Description embedding "white clothes / grey hair / asking for Ramkund" matched semantically with a pre-trained multilingual model — not trained from scratch, per the brief.
  • Duplicate detection same logic flags the 8% of records that are the same person across centers, cleaning the registry as it grows.
04

System design — offline & messy by default

The brief grades offline survival and dirty-data handling. We design for both first, not as an afterthought.

Offline-first, sync-when-you-can

  • Each console is a PWA backed by PouchDB in the browser. It works fully with zero signal.
  • PouchDB ↔ CouchDB (on a kendra box) replicate continuously — multi-master, revision-based conflict handling built in. No hand-rolled sync layer.
  • When signal returns, every console converges automatically; the rules engine scores on-device offline, and Claude re-scores the semantic/language fields once back online.
  • Fallback for total blackout: printed QR "case cards" a runner physically carries between kendras.

Messy / incomplete data

  • No required name or mobile — the matcher down-weights, never blocks. (15% no name, 20% no mobile in the data.)
  • Free-text descriptions normalized via the LLM into structured tags at intake.
  • Language is a signal, not a barrier: store original + English; match across both.
  • Duplicate-aware: 8% cross-center dupes are detected and merged, not double-counted.

PouchDB/CouchDB give real offline-first replication for free — the 2-tab "two centers" sync is the DB's native mode, not bolted on. A config-driven rules engine does the deterministic scoring on-device (always available, even offline). The LlmBackend is a swappable interface: Claude does the language-heavy lifting now (transcribe→translate→structure→explain a match, semantic description overlap); a local LLM drops into the same interface later for fully on-box, no-internet operation. No GPU, no model training.

4b

The AI is fully configurable

No scoring weight, threshold, or rule is hard-coded. Every knob lives in one config document in CouchDB that replicates to every console — edit once, applies everywhere, available offline. Both the rules engine and the LLM prompt read from it, so the system tunes without a redeploy.

What's tunable

  • Field weights — age, gender, state, description, location, time, name point values.
  • Confidence bands — HIGH / MEDIUM / LOW cutoffs + the minimum score to return.
  • Cluster rules — open-cases threshold & radius that mark a chokepoint zone "danger" (Mode C).
  • Escalation rules — child age ceiling, hours-open trigger, duplicate-count trigger.
  • Adjacency — which age-bands and zones count as "adjacent" for partial credit.

Why it matters here

  • One source of truth — rules engine (offline) and Claude (online) score identically; no drift.
  • Field-tunable — ops can dial sensitivity on a snan-day surge without a developer.
  • Auditable & responsible — the scoring policy is data, versioned in CouchDB, reviewable.
  • Future-proof — same config feeds the local-LLM backend when it replaces Claude.

The config doc (replicated to every console)

{
  "_id": "config:scoring", "version": 3,
  "weights":   { "age": 25, "gender": 20, "state": 20, "description": 20,
                 "location": 10, "time": 5, "nameExact": 10, "namePhonetic": 5 },
  "scoreCap":  100,
  "bands":     { "high": 75, "medium": 50, "low": 30 },
  "timeWindowHours": 6,
  "adjacency": { "ageBands": [["41-60","61-70"],["61-70","71-80"],["71-80","80+"]] },
  "cluster":   { "minOpenCases": 3, "radiusMeters": 300, "trafficChokepointsOnly": true },
  "escalation":{ "childAgeMax": 12, "childHours": 2, "unresolvedHours": 6, "dupReports": 3 },
  "llm":       { "backend": "claude", "model": "claude-opus-4-8",
                 "useFor": ["transcribe","translate","structure","semanticDescription","explain"],
                 "languages": ["en","hi","mr"] }
}

Swap "backend": "claude" → "local" in the future and nothing else changes. Offline, the rules engine reads weights/bands/adjacency directly; the description points are filled in by the LlmBackend when reachable.

05

Privacy by design

Missing-person data is among the most sensitive there is. We treat that as a feature constraint, not a checkbox.

  • Minimal collection — only what helps reunite. No Aadhaar, no caste, no permanent biometric store.
  • Found-person photos are ephemeral — used for the match window, auto-purged on reunion or after the mela. Never published to a public wall.
  • No public search — only authenticated volunteers query. Families get results through a person, removing the trafficking/scam risk of an open lost-children database.
  • Human in the loop on every reunion — the model proposes, a trained volunteer + ID check confirms before handover.
  • Consent & audit — every record logs who created/accessed it; data scoped to the event and deleted after.
  • Synthetic-only in the demo — we ship and present on the 2,500 fake records; zero real PII touched.

ALL

Console → registry → matcher → shortlist with plain-language reasons → confirm reunion. Make it work end-to-end with the network OFF, then sync back ON.

3:30–4:15
ALL

Privacy + polish + edge cases

Auth gate, photo auto-purge, audit log. Handle blank name / blank mobile gracefully on screen. EN/HI/MR labels finalized. Offline rules-only path verified (Claude unreachable → still scores). Empty/error states.

4:15–5:00
ALL

Deploy + stress test + rehearse

Ship to Vercel + CouchDB. Seed demo records. Offline stress test (network off → register 5 cases → re-enable → confirm all sync across both tabs). Spike test (loadAllData ×5 → search still <3s). Run the 3 demo scenarios live; cut a 90s backup video.

Cut-lines (if behind)

Drop in this order: hotspot prediction (Mode C) → voice (fall back to typed + language dropdown) → map polish → live config-editing UI (config doc still works, just edited by hand). Never cut: the cross-center match, offline + 2-tab sync working once, and the privacy story. Those three are the judging criteria. (Sync is no longer a cut candidate — CouchDB gives it for free.)
